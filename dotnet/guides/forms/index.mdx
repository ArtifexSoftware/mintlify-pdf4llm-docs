# Form field extraction

`PdfExtractor.GetKeyValues` extracts all interactive form fields from a PDF — their names, current values, and the page each field appears on. Use it to pull structured data from filled-in forms without parsing unstructured text.

---

## When to use this

Form field extraction is the right tool when:

- You have **filled-in AcroForm PDFs** — tax forms, applications, questionnaires, intake forms — and need the submitted values as structured data
- You want to **combine form data with document text** — extract form fields for structured storage and the surrounding text for RAG
- You are building a **document processing pipeline** that ingests completed forms and needs to route or validate their content
- You need to **identify which fields are on which pages** for spatial analysis or form mapping

It is not a substitute for `ToMarkdown` or `ToText` on general documents. Most PDFs do not contain interactive form fields — `GetKeyValues` on a report or a manual returns an empty list.

---

## Basic usage

```csharp
using MuPDF.NET;
using PDF4LLM;

Document doc    = new Document("application-form.pdf");
var      fields = PdfExtractor.GetKeyValues(doc);
doc.Close();

foreach (var field in fields)
{
    Console.WriteLine($"[Page {field.Page}] {field.Name}: {field.Value}");
}
```

### Example output

```
[Page 0] FirstName: Jane
[Page 0] LastName: Smith
[Page 0] DateOfBirth: 1990-04-12
[Page 0] EmailAddress: jane.smith@example.com
[Page 1] EmploymentStatus: Employed
[Page 1] AnnualIncome: 72000
[Page 1] SignatureDate: 2024-09-15
[Page 1] DeclarationCheckbox: Yes
```

## The FormField type

Each item in the returned list is a `FormField` object with three properties:

| Property | Type | Description |
|---|---|---|
| `Name` | `string` | The programmatic field name as defined in the PDF. |
| `Value` | `string` | The field's current value. Empty string if the field has not been filled. |
| `Page` | `int` | Zero-based page number where the field appears. |

Field names are set by whoever created the PDF form. They are often camelCase or PascalCase internal identifiers (`firstName`, `DateOfBirth`) rather than the human-readable labels visible next to the field on the page. Those visible labels are separate text elements — they are not part of the `FormField` object.

## Working with field values

### Build a flat dictionary

For most data extraction use cases, a dictionary keyed by field name is the most convenient structure:

```csharp
var fields = PdfExtractor.GetKeyValues(doc);

var data = fields.ToDictionary(
    f => f.Name,
    f => f.Value
);

Console.WriteLine(data["FirstName"]);    // Jane
Console.WriteLine(data["DateOfBirth"]); // 1990-04-12
```

<Callout type="warning">
`ToDictionary` throws `ArgumentException` if the same field name appears more than once — possible in some PDF generators. Use `GroupBy` with a safe merge if you expect duplicate field names:

```csharp
var data = fields
    .GroupBy(f => f.Name)
    .ToDictionary(g => g.Key, g => g.Last().Value);
```
</Callout>

### Filter by page

To process only the fields on a specific page:

```csharp
var page1Fields = fields.Where(f => f.Page == 1).ToList();
```

### Filter out empty fields

Unfilled fields return an empty string as their value. Exclude them when you only need submitted data:

```csharp
var filledFields = fields
    .Where(f => !string.IsNullOrWhiteSpace(f.Value))
    .ToList();
```

### Serialise to JSON

```csharp
using System.Text.Json;

var fields = PdfExtractor.GetKeyValues(doc);

string json = JsonSerializer.Serialize(
    fields.Select(f => new { f.Page, f.Name, f.Value }),
    new JsonSerializerOptions { WriteIndented = true }
);

File.WriteAllText("form-data.json", json);
```

Output:

```json
[
  { "Page": 0, "Name": "FirstName",        "Value": "Jane"       },
  { "Page": 0, "Name": "LastName",         "Value": "Smith"      },
  { "Page": 0, "Name": "DateOfBirth",      "Value": "1990-04-12" },
  { "Page": 1, "Name": "Signature",        "Value": ""           },
  { "Page": 1, "Name": "DeclarationCheckbox", "Value": "Yes"     }
]
```

## Common field types and their values

PDF AcroForms support several field types. `GetKeyValues` returns all of them as strings — the field type affects how the value is represented.

| Field type | Typical value representation |
|---|---|
| Text field | The entered text as-is |
| Checkbox | `"Yes"` when checked, `""` when unchecked |
| Radio button group | The value of the selected option (e.g. `"Option1"`) |
| Dropdown / combo box | The selected option value |
| List box | The selected option value; comma-separated if multi-select |
| Signature field | `""` — signature data is not extracted |
| Date field | The date string in whatever format the form uses |

Checkbox and radio button values depend on how the form was built — `"Yes"` and `"Off"` are common conventions, but some forms use `"true"` / `"false"`, `"1"` / `"0"`, or custom option names. Inspect a filled sample of your specific documents before writing parsing or validation logic that assumes particular values.

## Combining form fields with document text

Many workflows need both the structured form data and the surrounding document text. Since `GetKeyValues` and `ToMarkdown` both accept an open `Document`, run them together in a single open/close cycle:

```csharp
Document doc = new Document("application-form.pdf");

// Structured form data
var    fields   = PdfExtractor.GetKeyValues(doc);

// Full document text for RAG or storage
string markdown = PdfExtractor.ToMarkdown(doc);

doc.Close();

var record = new
{
    FormData     = fields.ToDictionary(f => f.Name, f => f.Value),
    DocumentText = markdown
};
```

This is the recommended pattern when ingesting completed forms into a system that uses both structured data — for filtering, routing, or database storage — and unstructured text for semantic search or LLM summarisation.

## Validating extracted data

`GetKeyValues` returns values as-is: no type coercion, no normalisation, no validation. That work belongs downstream.

### Check for required fields

```csharp
var data     = fields.ToDictionary(f => f.Name, f => f.Value);
var required = new[] { "FirstName", "LastName", "DateOfBirth", "EmailAddress" };

var missing = required
    .Where(name => !data.ContainsKey(name) || string.IsNullOrWhiteSpace(data[name]))
    .ToList();

if (missing.Count > 0)
    Console.WriteLine($"Missing required fields: {string.Join(", ", missing)}");
```

### Parse typed values

```csharp
// Date field
if (data.TryGetValue("DateOfBirth", out string? dob) && !string.IsNullOrWhiteSpace(dob))
{
    if (DateOnly.TryParse(dob, out DateOnly parsedDate))
        Console.WriteLine($"Date of birth: {parsedDate:yyyy-MM-dd}");
    else
        Console.WriteLine($"Could not parse date: '{dob}'");
}

// Numeric field
if (data.TryGetValue("AnnualIncome", out string? income) && !string.IsNullOrWhiteSpace(income))
{
    if (decimal.TryParse(income, out decimal parsedIncome))
        Console.WriteLine($"Annual income: {parsedIncome:C}");
}

// Checkbox — value depends on the form; inspect a sample to confirm
bool accepted = data.TryGetValue("AgreeToTerms", out string? agreed)
    && agreed?.Equals("Yes", StringComparison.OrdinalIgnoreCase) == true;

Console.WriteLine($"Terms accepted: {accepted}");
```

## Processing multiple forms

For bulk ingestion of a folder of completed forms:

```csharp
string folderPath = "./completed-forms";
var    allRecords = new List<Dictionary<string, string>>();

foreach (string pdfPath in Directory.GetFiles(folderPath, "*.pdf"))
{
    Document doc    = new Document(pdfPath);
    var      fields = PdfExtractor.GetKeyValues(doc);
    doc.Close();

    if (fields.Count == 0)
    {
        Console.WriteLine($"No form fields in {Path.GetFileName(pdfPath)} — skipping");
        continue;
    }

    var record = fields
        .GroupBy(f => f.Name)
        .ToDictionary(g => g.Key, g => g.Last().Value);

    // Track the source file
    record["_source"] = Path.GetFileName(pdfPath);
    allRecords.Add(record);
}

Console.WriteLine($"Processed {allRecords.Count} forms");
```

### Export to CSV

```csharp
// Collect all unique field names across all forms
var allFieldNames = allRecords
    .SelectMany(r => r.Keys)
    .Where(k => !k.StartsWith("_"))
    .Distinct()
    .OrderBy(k => k)
    .ToList();

var csvLines = new List<string>
{
    // Header row
    "Source," + string.Join(",", allFieldNames)
};

foreach (var record in allRecords)
{
    string source = record.GetValueOrDefault("_source", "");

    var values = allFieldNames.Select(name =>
    {
        string val = record.GetValueOrDefault(name, "");
        // Quote values containing commas or quotes
        return val.Contains(',') || val.Contains('"')
            ? $"\"{val.Replace("\"", "\"\"")}\""
            : val;
    });

    csvLines.Add($"{source},{string.Join(",", values)}");
}

File.WriteAllLines("form-export.csv", csvLines, System.Text.Encoding.UTF8);
```

## AcroForm vs XFA

PDF supports two interactive form specifications.

**AcroForm** is the standard format used by the vast majority of fillable PDFs — Acrobat-created forms, government forms, and forms produced by most PDF libraries. `GetKeyValues` fully supports AcroForm.

**XFA (XML Forms Architecture)** is an older XML-based format used primarily by Adobe LiveCycle Designer. XFA forms often look identical to AcroForms in a viewer but are stored very differently internally. `GetKeyValues` may return partial or no results for XFA-only PDFs.

To determine which format a form uses, inspect it in a PDF viewer that exposes form properties, or check the raw output from `ToJson` for the presence of AcroForm field annotations on the relevant pages. If your pipeline regularly encounters XFA forms, consult the [MuPDF.NET documentation](https://mupdfnet.readthedocs.io) for low-level XFA handling options.

## Troubleshooting

**`GetKeyValues` returns an empty list**
The document does not contain AcroForm fields. This is expected for the vast majority of PDFs — only documents explicitly created as fillable forms have interactive fields. Confirm the document is a fillable form by opening it in a PDF viewer and checking whether form fields are highlighted or selectable.

**All field values are empty strings**
The form exists but has not been filled in, or the PDF contains field definitions without submitted values. This is a valid state — an unfilled form returns all fields with `""` as the value. Distinguish between "form not found" (empty list) and "form found but unfilled" (list of fields, all with empty values).

**Field names are cryptic or numeric**
PDF field names are set by the form author and are not required to be human-readable. Names like `topmostSubform[0].Page1[0].TextField1[0]` are common in Adobe LiveCycle-generated forms. Build a static lookup table mapping internal names to readable labels using a sample form from your document set.

**Duplicate field names in the results**
Some PDF generators reuse field names across pages — particularly for running headers or repeated sections. Use `GroupBy` rather than `ToDictionary` when building a name-to-value map, and decide which occurrence to keep based on page order or field content.

**Checkbox values are not `"Yes"` / `"No"`**
Checkbox export values are set by the form author. Common alternatives include `"On"` / `"Off"`, `"true"` / `"false"`, and custom option names. Inspect a filled sample to determine the actual values used by your specific forms before writing boolean parsing logic.

**Signature fields always return empty string**
Signature fields store a cryptographic signature object, not extractable text. `GetKeyValues` returns `""` for all signature fields. Use the field's presence in the list and its `Page` property to confirm a signature block exists, but do not attempt to read or validate the signature through this API.

**Fields are present but appear on the wrong page**
`Page` is zero-based. Page 1 of the document is `field.Page == 0`. If your reporting uses one-based page numbers, add one when displaying: `$"Page {field.Page + 1}"`.
